# Daily Vehicle Usage Foundation

**Date:** 2026-04-18  
**Agent:** Gemini A (Backend Specialist)

## Overview
Implemented the backend intelligence layer for tracking and summarizing usage of the user's **Daily Vehicle**. This foundation calculates activity metrics and streaks based on odometer history, providing the data necessary for future gamification and usage-based insights.

## Changes
### 1. Usage Day Definition
- A "Used Day" is defined as any calendar day (UTC) where at least one valid `OdometerReading` exists for the current Daily vehicle.
- This rule is future-safe, as odometer readings can be generated by manual logs, service entries, or future Bluetooth/API integrations.

### 2. Usage Summary Service
- Added `getDailyUsageSummary(userId)` to `UserVehicleService`.
- **Metrics Computed:**
  - `currentStreak`: Consecutive days of odometer logging (including today and yesterday as valid "alive" states).
  - `lastLoggedAt`: Timestamp of the most recent odometer entry.
  - `activeDaysLast7`: Count of unique days with log activity in the trailing 7-day window.
  - `hasUsageData`: Boolean flag for graceful empty-state handling.

### 3. API Endpoint
- Added `GET /user-vehicles/daily/usage-summary`.
- **Response Shape:**
  ```json
  {
    "dailyVehicleId": "uuid",
    "hasDailyVehicle": true,
    "hasUsageData": true,
    "currentStreak": 5,
    "lastLoggedAt": "2026-04-18T12:00:00.000Z",
    "activeDaysLast7": 6
  }
  ```

## Why
By centralizing usage intelligence in the backend, we ensure a single, deterministic source of truth for "Activity Streaks" and usage patterns. This avoids client-side calculation drift and provides a consistent foundation for future push notifications and reminder triggers based on vehicle utilization.

## Validation
- **Deterministic Streaks:** Verified that streaks reset correctly if a day is skipped beyond the "yesterday" grace period.
- **Graceful Nulls:** Verified that users without a Daily vehicle or without log history receive valid, zeroed-out summary objects rather than errors.
- **Efficient Querying:** The service uses indexed date-sorted queries with limits to ensure high performance even for users with extensive log histories.

## Notes
- This system currently relies on `OdometerReading` history. Any future telemetry input that writes to this table will automatically feed the streak/usage system.
- Frontend visualization of the streak and activity summary is pending.
